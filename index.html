1. Context: What is Khmer Script?

🌏 Geographic & Linguistic Scope

Khmer script is the primary writing system for Cambodia, used by Central Khmer speakers and six ethnic minority languages (Bunong, Tampuan, Brao, Krung, Jarai, Kuay). Two classical languages — Pali and Sanskrit — also use Khmer script for Buddhist texts.

📝 Writing System Type: Abugida

Khmer is an Abugida — a writing system where each consonant carries an inherent vowel sound. Vowels are modified by adding signs around the consonant (above, below, before, or after).

Think of it this way: in English, "K" is just the sound /k/. In Khmer, the consonant ក (Ka) already means /kɔː/ — it has a vowel built in. You change the vowel by adding marks.

Direction: left to right, top to bottom. Origin: descended from Indian Brahmi script (5th-6th century CE).

⚠ Key Character: U+17D2 (Coeng)

The Coeng marker (U+17D2) is the most important functional character. It is not visible on its own — instead, it transforms the next consonant into a subscript (a small form rendered below or beside the base consonant). Grammar books list 32 subscripts, but Unicode generates them dynamically using U+17D2 + consonant.

💡 Key Insight

Visual position ≠ typing order. The vowel sign may appear to the left, right, above, or below the base consonant on screen — but in Unicode, it is always encoded AFTER the base and subscripts. This disconnect between what users see and what they should type is the root cause of most encoding errors.

Example 3: The Consonant Shifter (the tricky one)

The consonant shifter (U+17CA "Triisap" or U+17C9 "Muusikatoan") changes the register/series of a consonant. Here's the confusing part:

⚠ The Visual Trick

When U+17CA (consonant shifter) is followed by certain above-vowels, it changes its visual form to look like U+17BB (the below-vowel "O"). They are completely different characters, but look identical on screen.

Encoding: ស + U+17CA (consonant shifter) + U+17B8 (vowel) On screen: ស + shape that looks like U+17BB + above mark What users think they see: base + vowel-O + vowel-A What is actually encoded: base + consonant shifter + vowel-ii This visual trick causes users to type the WRONG characters because they type what they SEE, not what is LOGICALLY required.

Issue 1: Identical Renderings from Different Character Sequences

Different character orderings produce visually identical text. Users cannot detect the difference.

Issue 3: Anarchy in Typing — 13 Ways to Encode One Word

The word meaning "to detect" (/sɔɔp/) was found to have at least 13 different encodings, all rendering identically on Chrome 58 (Android 6.0.1).

CORRECT: 9F CA BE 94 (base + shifter + vowel + final consonant) All 13 variants (hex shorthand of last 2 digits): 1. 9F CA BE 94 ✓ 6. 9F BB BE 94 2. 9F CA C1 B8 94 7. 9F BB B8 C1 94 3. 9F CA B8 C1 94 8. 9F B8 BB C1 94 4. 9F BB C1 B8 94 9. 9F B8 C1 BB 94 5. 9F BE BB 94 10. 9F C1 B8 BB 94 11. 9F C9 C1 B8 94 12. 9F C9 B8 C1 94 13. 9F C1 BB B8 94 Some have been fixed in newer browsers (show dotted circle), but the incorrect data persists in texts and search indexes.
💡 Why does this happen?

Users type what they see, not what is logically correct. When the consonant shifter visually transforms to look like a vowel, users type two vowels instead of a shifter + vowel. And since the rendering engine displays them all correctly, there's no error feedback.

Issue 4: Security Risk — Spoofing Vulnerability

If Khmer script were used in Internationalized Domain Names (IDNs), the ability to create visually identical text with different encodings would enable domain spoofing attacks.

🚨 Attack Scenario
Legitimate domain: example.កា (correct encoding) Spoofed domain: example.កា (different encoding, looks identical) A user clicks the spoofed link thinking it's the real site. The browser shows the same visual domain name. The user has NO WAY to detect the difference.

This is similar to homograph attacks using mixed scripts (e.g., Cyrillic "a" vs Latin "a"), but potentially more severe because it uses the same script with different encoding orders.

The Formal Rules Side by Side

Unicode (2019:645): B {R | C} {S {R}}* {{Z} V} {O} {S} OpenType (2018): Cons {S}* V C V O V O Open Forum (2004): B R {V} {O} / B {S}* C V {O}* Where: B = base R = Robat C = consonant shifter S = subscript V = vowel Z = ZWJ/ZWNJ O = other sign {} = 0-2 [] = 0-1 * = 0+ | = or
⚠ The Biggest Conflict: Consonant Shifter Position

Unicode says: Base → Shifter → Subscript → Vowel

OpenType + Open Forum say: Base → Subscript → Shifter → Vowel

Unicode is the outlier — both other standards agree on placing the consonant shifter after subscripts.

💡 Why is the "intuitive wrong" at 37%?

Because users type what they see. The consonant shifter looks like a vowel on screen, so users type: base + vowel-O + vowel-A (what they see) instead of: base + consonant shifter + vowel-ii (what is logically correct). This is the single most common encoding error in Khmer text.

5. Proposed Solutions

Three complementary approaches to solve the problem:

🛡 Solution A: Rendering Engine Enforcement

What: Rendering engines should display a dotted circle (◌) when they encounter an illegal character sequence, instead of rendering it normally.

Why it helps: Users get immediate visual feedback that their input is wrong, prompting them to correct it.

Correct input: ស + U+17CA + U+17B8 → renders normally ✓ Incorrect input: ស + U+17BB + U+17B8 → ស◌ុ◌ី (dotted circles shown) The dotted circle tells the user: "Something is wrong here."
⌨ Solution B: Smart Keyboard Correction

What: The keyboard automatically reorders or converts characters as the user types. Users type naturally, and the keyboard ensures correct encoding output.

Why it helps: Zero learning curve for users — they type as they always have, but the data is always correctly encoded.

Implementation: The Khmer Angkor Keyboard (powered by Keyman, SIL International) already implements this. See the interactive demo below.

🤝 Solution C: Unify the Standards

What: Unicode Standard, OpenType, and Open Forum should adopt a single, unified character ordering rule.

Why it helps: Eliminates ambiguity for developers. All platforms would implement the same rules, ensuring consistent behavior across devices.

6. The New Proposed Rule

Proposed Unified Character Sequence
B ({S} | [R]) [C] [Z] [V] [O | S]
Current Unicode Standard:
B {R | C} {S {R}}* {{Z} V} {O} {S}

Why This Change?

1
Majority alignment: OpenType and Open Forum already use this order. Unicode is the sole outlier. 2 out of 3 standards already agree.
2
Simpler sorting: Subscripts directly modify the base consonant, so they should be adjacent for natural sort order: Base → Subscript → everything else.
3
Easier for developers: The proposed rule has a flatter structure with fewer nested groups — fewer edge cases to handle in code.
4
Backward compatible: The trailing [S] position accommodates archaic spelling where subscripts can appear after vowels.
5
Complete coverage: Handles all known Khmer character sequences — modern and historical.
